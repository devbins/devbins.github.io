关于代码质量整改的思考

前言

这段时间对代码质量进行整改，本篇文章以魔鬼数字为例，探讨了代码整改的目的，以及什么时候应该改，什么时候不应该改。

魔鬼数字

所谓的魔鬼数字指的是没有名字的数字，也就是直接在代码中写上具体的数字。如下所示

        break;
    case 1:
        // do something
        break;
        // ...
    default:
        // do something
}

上面的例子中数字就是魔鬼数字，你一眼看过去肯定不知道这些数字想要表达的是什么意思，这样的代码维护成本是非常高的，每个看到这段代码的人都可能会好奇每个数字代表的含义是什么。

如果有一天这些状态多了好几个，这对记忆也是个不小的负担。

魔鬼之所以是魔鬼是因为没有名字，像个鬼一样影响者代码的质量。

魔鬼数字在代码整改种还是非常常见的，这类问题刚开始整改的时候觉得挺麻烦的，直到我们钻了检测工具的空子就觉得简单起来了，但是后来发现这么做已经偏离了整改的目的，又觉得不简单了。

钻空子

最开始整改的时候，我们会对魔鬼数字取一个有意义的名字，但是这样发现取名字也并不是一件容易的事情，而且取了名字就意味着函数的行数增加，而我们对函数的行数是有限制的，这样改了可能就会导致另一个规则不通过。

于是我们发现了下面这种写法，检测工具不会检测出来（这里只讨论魔鬼数字的情况，数组的长度应该用一个有意义的名字来代替）

1

int prices[] = new int[10u];

于是魔鬼数字的整改就变成了在数字后面加上 u ，浮点数加上 f 。

在一段时间内，代码中充斥着大量的这种代码。我甚至觉得每次这么改太花时间了，于是想通过抽象语法树去检测，然后自动在魔鬼数字后面加上 u 、 f 之类的。这样还能节省不少时间。

不过就在研究方案的时候，我突然意识到似乎这样不对，因为魔鬼数字整改的本意是让我们取一个有意义的名字，提高代码的可读性，降低维护成本。

而这么做并没有提高代码的可读性，反而不美观，看着挺别扭的，有点多此一举。

这样做只是为了躲避检测工具的检测，忽略了整改的本意，走上了弯路。

经过这样的思考，我也就不再写工具去这么整改了，而是开始思考如何给魔鬼数字取一个有意义的名字。

整改过头

经过上面的讨论，魔鬼数字是影响代码质量的一个重要因素，是不是所有的魔鬼数字都是不好的？

魔鬼数字之所以不好，是因为魔鬼数字在某些情况下不能明确的表达它的含义，给代码的维护带来了巨大的负担。

然而魔鬼数字在某些情况下，它自己足够表达它的含义，这时候它就不再是魔鬼了，来看下面这个例子

1
2
3
4
5

arr[0];
arr[1];
arr[2];

int center = width / 2;

假如我要从数组中取第 1 个元素， arr[0] 可以非常好的表达它要取第 1 个元素，这时候它就不再是魔鬼数字了。

又比如我要取宽度的中点，只需要对宽度除以2，这个 2 已经非常明确要表达我要传递的意思，通常我们不会认为类似这种代码是魔鬼数字。

但是事情往往没这么简单，由于你所提交的代码会有工具去检测有没有违反规则，这时候你提交了魔鬼数字，你的代码就不能合入。

你可能想取个 one 、 two 、 three 来代替，这种做法也是不被允许的。所以为了应付检测工具，还得想一些歪招，比如上面的在数字后面加 u 、 f 之类的。要不就取一个别扭的名字，可读性反而降低了。

这种做法属于矫枉过正，改过头了，检测工具只能简单的对数字进行匹配，无法作出有效的判断，这种情况建议还是不要为了 KPI 进行改动。

有时候魔鬼也是天使，取决于具体的场景。不是所有的情况数字都是魔鬼，有时候也是天使。

总结

  1. 我们思考了魔鬼数字的危害
  2. 理解了整改魔鬼数字的本意
  3. 辩证的看待魔鬼数字
  4. 魔鬼之所以是魔鬼是因为没有名字
  5. 有时候魔鬼也是天使，取决于具体的场景，不要一棍子打死，需要辩证的看待。
